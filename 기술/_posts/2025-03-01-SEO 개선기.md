최근, 봄봄의 구글 검색 게재 순위가 불과 하루만에 폭락한 문제를 겪었습니다.  
이번 글은 검색엔진 결과에서 서비스가 노출되지 않는 문제를 파악하고 해결한 과정을 소개하려 합니다.

[봄봄](https://www.bombom.news/)은 Google Search Console에서 SEO 지표를 확인하며 관리하고 있습니다.   

특히 저희 팀원들은 SEO를 중요하게 생각하며 최소 일주일에 두 번은 주기적으로 모니터링하고 있습니다.

마케팅 비용은 한정되어 있고, '뉴스레터'라는 도메인의 특성상 뉴스레터를 읽는 사람들이 '뉴스레터'를 검색하고 유입될 가능성이 큽니다.

따라서 저희는 **검색 결과의 상위에 노출되는 것이 최소한의 비용으로 최고의 효율을 낼 수 있는 마케팅**이라고 생각하고 있습니다.

<br>

---

## 문제 상황

지난 2월 26일, 봄봄의 구글 검색 순위가 하루 만에 크게 떨어졌습니다.

![graph_marked.png]

처음에는 단순한 검색엔진 변동일 가능성도 생각했습니다.  
하지만 메인 페이지 URL을 점검하던 중 예상치 못한 사실을 발견했습니다.

---

## 1. URL 테스트에서 `noindex`가 확인됐습니다

메인 페이지를 테스트했더니, 인덱싱이 `noindex` 태그에 의해 차단되고 있었습니다.

![noIndex.png]

그런데 이상한 점이 있었습니다.  
실제 메인 페이지(`/`)는 원래 `index, follow`로 설정해둔 상태였기 때문입니다.

즉, **설정상으로는 인덱싱이 가능해야 하는데**, 검색엔진이 읽은 결과에서는 `noindex`가 보이고 있었습니다.

---

## 2. 크롤링된 HTML을 보니 `<head>`와 `<body>`가 섞여 있었습니다

이상함을 느끼고 실제로 크롤링된 HTML을 분석해보았습니다.  
그 결과, 메인 페이지와 랜딩 페이지의 정보가 하나의 문서 안에 섞인듯한 형태가 확인되었습니다.

대표적으로 확인된 특징은 다음과 같았습니다.

- `<head>` 상단에는 `/` 기준 메타 정보가 들어 있었습니다.
- 동시에 앱 번들 스크립트가 포함되어 있었습니다.
- `<body>`에는 `/landing` 페이지가 렌더된 결과가 들어 있었습니다.
- `<head>` 안에는 `robots=index,follow`와 `robots=noindex,follow`가 함께 존재하고 있었습니다.

예를 들면 다음과 같은 형태였습니다.

```html
<!doctype html>
<html lang="ko">
  <head>

    ...

    <meta name="robots" content="index, follow" />

    ...

    <meta name="robots" content="noindex, follow" />

  </head>
</html>
```

처음 보면 마치 서버가 HTML 파일 두 개를 이어붙여서 잘못 내려준 것처럼 보입니다.

하지만 분석을 진행할수록, 이 현상은 단순한 파일 병합 문제가 아니라  
**SPA의 초기 HTML과 JavaScript 실행 이후의 렌더 결과가 한 문서 안에 함께 반영된 결과**로 보는 것이 더 정확하다고 판단했습니다.

---

## 서비스 구조: 왜 이런 현상이 가능할까?

저희 서비스의 메인페이지는 다음과 같은 동작을 수행하고 있었습니다.

- 사용자가 처음 `/`에 접속합니다.
- 클라이언트 실행 후 `/landing`으로 리다이렉트됩니다.

즉, **서버는 `/`에 대한 초기 HTML을 내려주지만**,  
**브라우저나 크롤러가 JS를 실행한 이후에는 `/landing` 기준의 화면과 메타 정보가 반영될 수 있는 구조**였습니다.

---

## 왜 이런 HTML이 만들어졌을까요?

핵심은 Googlebot이 **초기 HTML만 보는 것이 아니라, JavaScript 실행 이후의 렌더 결과도 함께 분석한다는 점**입니다.

Google Search Central 문서에 따르면, Googlebot은 JavaScript 페이지를 대략 다음과 같은 흐름으로 처리합니다.

1. URL 크롤링
2. HTML 파싱 및 링크 발견
3. 렌더링 큐 등록
4. JavaScript 실행
5. 렌더된 HTML 재분석
6. 인덱싱 및 canonical 판단

즉, 크롤러는 단순히 서버가 처음 내려준 HTML만 읽고 끝나는 것이 아닙니다.  
**초기 HTML을 본 뒤, JS 실행 후 바뀐 결과까지 다시 반영해 판단할 수 있습니다.**

---

## 왜 테스트 URL과 실제 인덱싱 결과가 달라 보였을까?

이 지점에서 의문이 생깁니다.

> 분명 `/`을 검사했는데, 왜 결과는 `/landing` 콘텐츠가 섞여 보일까요?

이것은 SPA의 동작 방식을 생각하면 설명할 수 있습니다.

<br>

**실제 흐름은 다음과 같습니다.**

1. `/` 요청이 들어오면 서버는 기본 HTML 셸을 반환합니다.
   - 여기에는 기본 `<head>` 정보가 포함되어 있습니다.
   - 예:
     - `<title>봄봄</title>`
     - `robots=index, follow`

2. 이후 JavaScript가 실행되면서 React Router가 동작합니다.

3. 앱이 `/landing` 라우트를 렌더링하면:
   - `<body>`는 랜딩 페이지 콘텐츠로 바뀌고
   - `<head>`에도 랜딩 페이지용 메타가 추가됩니다.
   - 예:
     - `<title>봄봄 | 읽고 남기고 쌓는 뉴스레터 리딩 플랫폼</title>`
     - `robots=noindex, follow`

결국 검색엔진 입장에서는,

- 처음에는 `/`의 `<head>`를 읽었고
- 이후 JS 실행 결과로 `/landing`의 `<body>`와 추가된 `<head>` 정보를 보게 된 셈입니다.

그래서 최종적으로는 **두 HTML 파일이 합쳐진 것처럼 보이는 문서**가 관찰됩니다.  
하지만 실제로는 파일을 이어붙인 것이 아니라, **하나의 SPA 문서가 로딩 과정에서 순차적으로 변경된 결과**에 가깝습니다.

---

## 왜 어떤 시점에는 정상이고, 어떤 시점에는 문제가 생겼을까요?

이 현상이 더 헷갈렸던 이유는, 같은 구조여도 검색 결과가 항상 똑같이 나타나지 않았기 때문입니다.

즉, 아무 수정이 없었는데도 어떤 시점에는 비교적 정상적으로 보였고 또 어떤 시점에는 검색 순위가 크게 흔들렸습니다.

이 역시 Google의 처리 방식으로 설명할 수 있습니다.

### 1. 크롤링과 렌더링은 분리된 단계입니다

Google은 먼저 `/`의 원본 HTML을 수집하고 그 이후 별도의 렌더링 단계에서 JavaScript를 실행해 최종 상태를 다시 분석합니다.

따라서 어떤 시점에는 `/`의 원본 신호가 더 강하게 반영될 수 있고 또 다른 시점에는 JS 실행 후 `/landing` 결과가 더 많이 반영될 수 있습니다.

### 2. canonical과 sitemap 신호가 충돌할 수 있습니다

또 하나의 문제는 대표 URL 신호가 명확하지 않았다는 점입니다.

- `/landing`이 sitemap에 포함되어 있다면 검색엔진은 이를 주요 URL 후보로 인식할 수 있습니다.
- 그런데 `/landing`은 동시에 `noindex` 상태였습니다.
- 반면 `/`은 index 가능 상태였지만, 렌더링 후 결과에서는 `/landing`의 정보가 섞여 들어왔습니다.

즉, 검색엔진 입장에서는 다음과 같은 혼란이 생길 수 있습니다.

- `/`을 대표 페이지로 봐야 할까?
- `/landing`을 대표 페이지로 봐야 할까?
- 그런데 `/landing`은 `noindex`인데 왜 렌더링 결과에는 계속 보일까?

이처럼 **canonical, sitemap, robots 신호가 서로 일관되지 않으면**  
Google이 대표 URL을 재평가하거나, 예상과 다른 인덱싱 판단을 내릴 수 있습니다.

---

## 정리: 이번 문제의 핵심 원인

결론적으로 이번 문제의 핵심은 아래 두 가지였습니다.

### 1. `/`가 완성된 정적 HTML이 아니라 SPA 진입 셸이었습니다

검색엔진은 `/` 요청 시 기본 HTML을 먼저 받았지만,  
그 문서는 최종 완성 페이지가 아니라 **클라이언트 앱이 동작하기 전의 진입점**이었습니다.

### 2. 클라이언트 실행 후 `/landing`으로 이동하는 구조였습니다

JavaScript가 실행되면 앱이 `/landing`으로 이동했고,  
그 결과 `<body>`는 랜딩 페이지로 바뀌고 `<head>`에는 랜딩 페이지용 메타가 추가되었습니다.

즉, 검색엔진 입장에서는 `/`의 초기 문서와 `/landing`의 렌더링 결과가  
한 문서 안에 함께 존재하는 것처럼 보일 수밖에 없는 구조였습니다.

---

## 최종 결론

이번 현상은 서버가 HTML 파일 두 개를 잘못 합쳐서 내려준 문제가 아니었습니다.

실제로는,

- `/` 요청 시 반환된 **초기 SPA HTML 셸**
- JavaScript 실행 이후 반영된 **`/landing` 라우트의 렌더링 결과**

이 두 단계가 검색엔진의 분석 과정에서 함께 반영되면서,  
결과적으로 **`<head>`와 `<body>`가 섞인 것처럼 보이는 HTML**이 관찰된 것입니다.

그리고 이 과정에서 `robots=index`와 `robots=noindex`가 동시에 노출되면서,  
검색엔진이 페이지의 대표성과 인덱싱 가능 여부를 일관되게 해석하기 어려운 상태가 만들어졌습니다.

---

## 배운 점

SPA 환경에서 SEO를 다룰 때는 단순히 “초기 HTML에 메타 태그를 넣었다”만으로 충분하지 않습니다.  
실제로는 아래 조건들이 모두 일관되어야 합니다.

- 서버가 반환하는 초기 HTML
- JS 실행 후 최종 렌더링 결과
- route별 meta tag 주입 방식
- canonical 설정
- sitemap에 포함된 URL
- robots 정책

특히 **초기 URL과 최종 렌더링 URL이 다르고**,  
그 과정에서 **`<head>` 정보까지 동적으로 바뀌는 구조**라면  
검색엔진은 우리가 예상한 것과 전혀 다른 형태로 페이지를 해석할 수 있습니다.

이번 이슈는 단순한 `noindex` 실수가 아니라,  
**SPA 렌더링 구조와 검색엔진 처리 방식이 충돌하면서 생긴 문제**라는 점에서 더 의미 있는 사례였습니다.
